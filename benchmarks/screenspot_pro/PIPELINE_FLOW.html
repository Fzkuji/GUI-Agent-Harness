gui_harness/planning/screenspot_locator.py

Harness 内部流程 · 迭代缩放定位

screenspot_locate() 在 pipeline=harness 的模型(目前只有 GPT-5.5)上 实际跑的流程——每一个方框都是一次独立的模型调用,分支就是代码里真实存在的判断逻辑,不是简化示意。

输入

截图 + 候选证据

全图截图 img_w × img_h

GPA-YOLO + OCR 检测(detect_components)产出候选框, 经 build_candidates 合并为证据列表,随每轮裁剪一起喂给模型。

↻ 循环 · 最多 iterative_rounds = 8 轮,提前 final 可跳出

① 裁剪决策 crop-decision

模型看当前裁剪 + 候选证据(裁剪内/外)+ 历史决策,输出 action。

action = crop
提交提议裁剪框 → 进入 Commit Gate
action = final
提前跳出循环 → 直接进入 Final Click
action = recrop
退回更宽裁剪框,本轮重来

② Commit Gate 仅 action=crop 触发

单独一次调用审核提议的裁剪框——是否包住目标候选簇、是否留够上下文、 有没有丢掉同任务的竞争候选。

accept
提交新裁剪框(面积达标:stage1≥20% / stage2≥8%)→ 进入下一轮
reject
按 crop_check_reject_mode 退到更宽范围(widen) 或全图重来(restart_full),重试 ≤ crop_retry_limit(3)次

轮间记忆:文字一直都有,图片默认不留

三层是独立开关,不是自动打包在一起——GPT 生产配置(sspro_stack_zoom.yaml)只开了第一层。

压缩文字摘要(最近 8 轮)
_format_iterative_history:每轮 action / crop_box / next_box / area_fraction / target_visible_element 各一行, 塞进下一轮的 prompt。模型知道“自己判断过什么”,但看不到那几轮的画面。
始终开启
完整对话线程 context_mode=accumulate
把每个被接受轮次(gate 拒绝/recrop 不算)发送的完整文本块 + 模型原始回复, 滚雪球式累加进后续轮次——比上面的摘要保留细节多得多,但只是文字。
GPT 现用:关(single)
把裁剪图也带上 accumulate_images=true
需先开启上一层,再叠这一层:之前被接受轮次的裁剪图会一起留在上下文里, 模型能“回看”走过的每一张图,而不只是文字摘要——但代价是每轮上下文体积线性增长。
GPT 现用:关

阶段二

Final Click

渲染最终裁剪,一次调用给点

(可选 enable_final_candidate_detect)在最终区域重新检测候选 → 一次调用给 x,y 或 candidate_id → 映射回原图坐标 (cx, cy)。

enable_final_recheck
额外一次 Review 调用复核:keep(保留)/ replace(替换点)
点击无效 / 模型要求 recrop
enable_final_recrop 退回更宽裁剪, 重试 Final Click ≤ final_recrop_limit(3)次

Exhaustion Fallback 重试全部耗尽仍无有效点击

exhaustion_fallback = none
返回 None — 该题判错
exhaustion_fallback = keep_best
返回最深一次已提交裁剪的中心点(低置信度兜底,好过完全没有答案)
{"cx": ..., "cy": ..., "grounding_type": "iterative_zoom_crop_refine", "confidence": ..., "reasoning": "..."}
为什么算作 ~15–20 次调用:最多 8 轮裁剪决策,每轮命中 reject 时 commit gate 最多重试 3 次;加 1 次 final click,最多重试 3 次 recrop;可选 1 次 recheck。—— 比 native single-shot 的 1 次调用贵一个数量级。这正是它对 GPT 有净增益(+10.4pt)、却对 qwen 是净负担(−8pt)的原因: 候选约束 + commit gate 对弱定位模型是“扶一把”,对强定位模型是“捆住手脚”。
