# ScreenSpot-Pro 坐标格式与"每模型策略"调研

> 核心发现:**视觉定位存在两种性质完全不同的能力——一种是监督训练出的格式绑定特化技能
> (一次前向直接映射坐标,不依赖推理,换个等价表达就近乎失效,天花板由训练数据分布固定死);
> 另一种是通用视觉推理能力(能被证据、多轮修正、脚手架持续放大,不挑坐标表达方式)。我们的
> 方法(证据脚手架+迭代裁剪)要挖掘和放大的是后者,而不是前者——坐标格式敏感性本身就是区分
> 这两种能力的诊断信号,本文提出的三探针诊断法(thinking敏感性/证据敏感性/证据符号一致性)
> 加上一个更直接的格式鲁棒性指标(同一模型在4种坐标格式下原始准确率的方差,方差小才是真鲁棒)
> 就是用来做这件事的。**

日期:2026-07。基准:ScreenSpot-Pro(1581 题,判分=预测点落在 GT 框内,无 IoU)。
评测子集:`baseline50`(从 m3_zoom 300 子集等距抽 50,harness 逐样本成绩现成);
`qwen 504`(阿里云主跑抽 120 配对)。所有单发探针=一次 API 调用/题。

---

## 1. 缘起

qwen3.7-plus 套本 harness(迭代缩放 sspro_stack_zoom.yaml)在 SSPro 全量 62.9%,明显低于
GPT-5.5 同 harness 的 87.9%,也低于 Qwen 官方自报的 79%。逐步排查发现根因是**坐标格式**,
而非模型能力、也非分辨率。

## 2. 关键发现:坐标格式是决定性变量,且模型相关

### qwen3.7-plus 单发格式消融(baseline50,同 ~48 题,无提示)

| 格式 | 正确率 | 说明 |
|---|---|---|
| 绝对像素(缩放图空间) | 16% | qwen 最差 |
| [0,1] 浮点分数 (default) | 52% | |
| [0,1] 浮点分数 (+hires) | 58% | |
| **{x,y} 整数 [0,1000]** (+hires) | **69%** | scale 对了 +11 |
| **point_2d 整数 [0,1000]** (+hires) | **79%** | 再 +10,**精确复现厂商自报 79.0** |

结论:qwen 训练用 **[0,1000] 归一化整数 + `{"point_2d":[x,y]}`**。喂 [0,1] 分数或绝对像素
白丢最多 21 分(58→79)。拆解:[0,1000] scale +11、point_2d 包装 +10。单向、n=48 远超噪声。

### GPT-5.5 单发格式消融(baseline50,无提示)

| 格式 | 正确率 |
|---|---|
| **绝对像素** | **62%**(最高) |
| {x,y} / point_2d [0,1000] | 56%(−6) |
| [0,1] 分数 | 50% |

结论:GPT(OpenAI,像素/computer-use 训练)偏好**绝对像素**,所有归一化格式都更差。

### 两模型完全镜像

| | 绝对像素 | [0,1000]+point_2d |
|---|---|---|
| **GPT-5.5** | **62% ✅** | 56% |
| **qwen3.7-plus** | 16% | **79% ✅** |

**没有统一坐标格式,甚至完全相反。**

### 2.5 真正的格式鲁棒性:同一模型跨4种格式的原始准确率方差(2026-07,四模型补全后回算)

之前把"提示效应会不会跟着格式反转"(见 §9.6 三探针)也叫"格式鲁棒性",这个命名不准确——那测的是
证据效应的**符号**,不是鲁棒性本身。**真正的格式鲁棒性应该看:不管用哪种坐标格式问同一个模型,
它的原始单发准确率本身稳不稳(方差小=鲁棒)**,这个数据其实上面 §2/§7.5/§9.6 的消融里已经有了,
回算标准差:

| 模型 | 4种格式准确率 | 标准差 | 鲁棒性 |
|---|---|---|---|
| **GPT-5.5** | 62/56/56/50 | **4.2pt** | **最鲁棒**——换格式几乎不影响性能 |
| MiniMax-M3 | 0/22/24/27 | 10.7pt | 中等 |
| kimi-k2.6 | 19/60/44/36 | 14.8pt | 中等偏差 |
| qwen3.7-plus | 16/58/69/79 | **24.0pt** | **最不鲁棒**——从16%到79%,极度依赖单一格式 |

这个指标比三探针的符号反转更直接、更容易理解:qwen 的标准差是 GPT 的近 6 倍,直接证明它的坐标
输出能力被死死绑定在一种训练约定上,换个等价问法就近乎失灵——这是"格式绑定特化技能"最朴素的
证据,不需要额外做 hint/thinking 实验就能看出来。

## 3. OCR/YOLO 提示:不是模型无关的杠杆——是格式相关的(⚠️ 已推翻旧结论)

单发注入检测候选(纯文本,零裁剪),**在旧的次优坐标格式下**测出:GPT +16(58→74)、
qwen +26(16→42,abs 格式下)/+6(54→60,frac01 格式下)。曾经把这个当作"模型无关的正向杠杆"。

**这个结论错了,已在 qwen 的真实最优格式(point2d_1000+hires)上重测,方向反转**:

| 配置 | 正确率 |
|---|---|
| 无提示 | **77%**(37/48) |
| 有提示 | 60%(29/48,**−17pt**) |

配对检查排除噪声:同 48 题,9 题被提示拖垮(本来对、加了提示后错)、仅 1 题被提示救回,
9:1 明显偏向一边。**qwen 在它的原生格式下已经是很强的单发 grounder,检测候选(本身不完美,
还占提示篇幅)反而在干扰它,不是帮它。**旧的"+26/+6"是在次优格式下测的假象——格式本来就弱,
任何拐棍都显得有用;真正强的格式不需要拐棍。

GPT 那边的 +16 不受影响——GPT 的提示消融本来就是在它的真实最优格式(绝对像素)上测的。

**铁律更新:`use_hints` 必须在锁定 `coord_format` 之后重测,不能沿用在其他格式下测出的数字。**
`model_profiles.py` 已更新:qwen `use_hints=False`(已验证);kimi `use_hints=False` 是保守默认,
**从未测试过**(kimi 的 4 格式消融全部是无提示条件跑的),需要专门补测。探针见
`probe_qwen_hints_on_best.py`。

**没有止步于此——专门排查过是不是"候选质量差"导致的假阴性。** 之前的提示是 120 条检测器
原始顺序、未排序未去重的堆砌;换成 harness 生产代码里已验证的 `candidate_sort="relevance"`
排序逻辑(`_candidate_relevance`,按指令关键词/OCR 类型打分)+ 精简到 top-12 重测:

| 版本 | 正确率 |
|---|---|
| 无提示 | 77% |
| 原始堆砌(120条,未排序) | 60% |
| **relevance 排序 + top-12** | **65%(比原始堆砌 +5,但仍比无提示 −12)** |

配对:relevance-hint vs 无提示,7 题被拖垮、仅 1 题被救回(7:1),依然单方向、非噪声。
**结论:候选质量确实是部分原因(排序后从 −17 收窄到 −12),但不是全部——即便用生产级排序
+精简,候选证据仍然拖累 qwen。这不是我们提示做得糙,是 qwen 在其原生格式下的直接感知本身
就比任何外部候选证据更可靠。** 探针见 `probe_qwen_hints_relevance.py`。

## 3.5 thinking 开关:另一个独立的负向变量(排除了它是提示假阴性元凶的可能)

一直没显式控制过 `enable_thinking`(阿里云端点参数),用的是端点默认值。怀疑"提示拖累 qwen"
是不是因为 thinking 状态没控制住的假象——在 22 题难样本子集(之前提示条件下有分歧的题)上做
hints×thinking 的 2×2:

| | thinking 关 | thinking 开 |
|---|---|---|
| 无提示 | **64%**(14/22) | 45%(10/22) |
| relevance 提示 | 32%(7/22) | 14%(3/22) |

两个变量**独立起作用、互不解释**:提示这条轴不管 thinking 开关都是负的(−32/−31,几乎相等)——
**排除了"thinking 混淆了提示效应"这个假设**。但意外收获:**thinking 本身也是负的**,而且降幅
比提示还大(平均 48%→30%)。呼应 Qwen 官方 79.0 就是 thinking-off 测出来的。**已把
`run_sspro_native.py` 的 `_make_aliyun_call` 固定为 `enable_thinking=False`**,四组里最优的
"无提示+thinking关"(64%)现在是生产默认配置。探针见 `probe_qwen_hints_x_thinking.py`。

## 4. 分辨率:约 2–4M 饱和,非瓶颈

qwen 单发自控 smart_resize:max_pixels 1M=45% → 4M=52% → 8M≈52%(饱和)。发完整 4K 原图
已达 ~4M 等效。`vl_high_resolution_images=true` 另给约 +6(52→58,处噪声地板,可选)。
**qwen 60% 不是分辨率饿出来的。**

## 5. qwen 的高分是特化技能的天花板,不是我们方法要争的目标

同 120 题配对(qwen 504 主跑抽样):

| 方法 | 正确率 | 成本 |
|---|---|---|
| **原生单发 [0,1000]+point_2d+hires** | **78%** | 1 次调用 |
| 现有 harness(绝对,多轮) | 70% | ~15–20 次调用 |

原生 **+8 反超** harness,还只用 1/20 调用、快 10–20 倍。仅原生对 19 / 仅 harness 对 9 /
**并集 86%**(有融合余量)。

原因:qwen 是**强原生 grounder**(专训 [0,1000] 定位),YOLO 候选约束 + 多轮裁剪反而碍事。
把 [0,1000] 强塞进 harness 还会和 crop-decision 的**像素框架打架** → commit gate 疯狂拒 →
单题 10–50 分钟爬不动(故放弃"把整个 harness 坐标框架改 [0,1000]"的大重构)。

## 5.5 追问:候选提示能不能救回来?——五连测,五连负

发现"候选提示拖累 qwen"后没有直接认输,而是系统排查了每一种可能的"是不是我们做糙了":

| 尝试 | 做法 | 结果 vs 无提示基线 |
|---|---|---|
| 原始堆砌 | 120条,检测器原序,未排序未去重 | 60%(**−17**) |
| relevance 排序+精简 | 复用 harness 生产的 `candidate_sort="relevance"`,top-12 | 65%(**−12**,配对7:1单向) |
| OCR 单独 | 只给 OCR 文字候选 | 45%(**−19**,22题难样本) |
| detector 单独 | 只给 icon 检测候选 | 45%(**−19**,22题难样本) |
| 模型自选裁剪两轮精修 | 一次调用同时要 `point_2d`+`bbox_2d`,按模型自己选的框裁原生分辨率精修一次 | 54%(**−14**,配对13:6被弄错) |

**五种独立方向,全部是净负。** 关键诊断:2 轮精修不是机制/解析问题(50题 `bbox_2d` 全部正常解析),
是**"再看一眼"这个动作本身系统性地让 qwen 推翻自己本来正确的判断**(13 题被精修弄错,仅 6 题被
救回)——跟候选提示拖累它是同一种效应。探针:`probe_qwen_hints_relevance.py` /
`probe_qwen_hints_split.py` / `probe_qwen_2round_modelcrop.py`。

## 5.6 为什么会这样:专训预测头 vs 通用推理(战略解释)

**假设**:qwen 的 `point2d_1000` grounding 是监督微调出来的专用能力,更接近"坐标回归头"而不是
"靠推理解题"——一次前向直接映射图像到坐标,不是生成式的逐步推理。三条独立证据支持:

1. **thinking 开关是反的**:22题难样本 2×2,thinking 关的两组均值 48% > thinking 开 30%
   (§3.5)。如果 grounding 靠推理,想得越久该越准;实测相反,说明它根本不是靠推理链条完成的。
2. **"再想想"类操作全部负**(候选证据、二次精修,共五个独立实验,见 §5.5)——你没法跟一个
   回归头"讲道理"、喂它证据或让它复核,那不是它工作的方式。
3. **逼它用没专训过的格式,提示效应直接反转**:同一批题,只换坐标格式:

   | 坐标格式(离专训约定的距离) | 无提示 | 有提示 | 提示效应 |
   |---|---|---|---|
   | abs_pixel(完全没专训过) | 16% | 42% | **+26pt** |
   | frac01([0,1]分数,半对齐) | 54% | 60% | +6pt |
   | point2d_1000(专训原生格式) | 73~77% | 60~65% | **−12~−17pt** |

   格式离专训约定越远,提示越正向;越接近,提示越负向——几乎单调。逼 qwen 用弱格式,相当于
   把它推出"预测通路"、拽进"通用推理通路",这时候证据才重新变得有用(但天花板仍只有 42%,
   远低于用对格式的 73~77%,所以这只是机制验证,不是更优方案)。

**战略结论**:视觉定位存在两种性质完全不同的能力——**格式绑定特化技能**(qwen 的
`point2d_1000`)一次前向直接映射,天花板由训练数据分布固定死,工程手段拿不动;**通用视觉推理
能力**(GPT 展现出的模式)能被证据、多轮修正、脚手架持续放大,且不挑坐标表达方式。**我们的方法
要挖掘和放大的是后者**——这才是能迁移到新应用、新界面、新指令的真实能力,不是在某个 benchmark
的坐标约定上"背对了答案"。qwen 单发领先 harness,与其说是"qwen 更强",不如说是**这道题对它而言
不再是一个视觉推理问题**,我们的方法自然帮不上忙——这是诊断法要识别出来的信号,不是方法的短板。
**未来的 harness 工程精力应该集中在像 GPT、M3 这样能被我们的方法持续拉高的通用推理型模型上,
qwen 这类特化捷径已经收口,不会再有回报。** 新模型接入时,用现有 SOP(格式消融 + native vs
harness 配对 + 三探针)先判定它是哪一类,再决定要不要投入。

## 6. 每模型最优策略(相反)

| 模型 | 坐标格式 | 要不要脚手架 | 最优分 |
|---|---|---|---|
| **GPT-5.5** | 绝对像素 | **要** harness(迭代裁剪) | **88.7%**(单发 78.3%⚠️,见 §9 审计) |
| **qwen3.7-plus** | [0,1000]+point_2d | **不要**,原生单发即可 | **73~78%**(harness 仅 70%) |
| 参照 M3 | — | harness | 47.4% |

**通则:每个模型不仅坐标格式不同,连要不要脚手架都不同。新模型上线前必先做便宜的单发格式
消融,找到其原生坐标格式,否则可能像 qwen 一样白丢 20 分而不自知。**

⚠️ **GPT 单发 78.3% 可靠性存疑,已在 §9 审计中查明**:它来自 `runs/sspro_stack/single/`(300题
旧数据),`grounding_type` 是 `direct_pixel`,追到代码发现这其实是 `active_loop`(候选池+轮次
决策)机制在没选中具体候选时的兜底分支,**不是纯净单发调用**,且产出它的确切旧代码路径已找不到
对应版本。GPT harness 88.7% 本身经核实是干净的(299/300 为当前版 `iterative_zoom_crop_refine`),
不受影响;但"harness 净赚 +10.4pt"这个具体差值,精确度不如原先讲的有把握。真正逐字验证过 prompt
的 GPT 干净单发数据是 `probe_ablation_gpt.py` 的 A 组(58%,50题,§2),但那是另一批题、不能直接
对比 88.7%。**若要坐实这个差值,需要在同一 300 题上用确认干净的方式重测 GPT 单发。**

## 7. 关于 79% 的出处(诚实说明)

Qwen 官方自报 ScreenSpot-Pro 79.0(thinking 关闭、方法未公开)。llm-stats 榜列 qwen3.7-plus
0.790(第4),但页面标注 **"23 self-reported results, 0 verified"——全部厂商自报、零第三方
复现**(Claude 0.879 / GPT 0.863 同样自报)。我们用官方 [0,1000]+point_2d 格式在 baseline50
上**独立复现到 79%**,确认该数在正确格式下真实可达;之前的 52–60% 纯属格式错配。

⚠️ **可靠性说明(§9 审计发现)**:这个 79% 的原始文件(`qwen_official.jsonl`)后来被一次脚本
误操作(sed 替换写错输出路径)意外覆盖损毁,现在没有可重新核验的原始文件——只有当时的会话
记录(控制台输出 `38/50=79%`)和紧接着做的配对分析(净+10)。**但同一配方
(point2d_1000+hires+无提示)后续被独立重跑了至少两次**:`qwen_hints_on_best_nohint.jsonl`
(77%,48题)、`qwen_best_with_coords.jsonl`(73%,50题,带坐标),数字都落在 73~79% 区间,
互相印证,结论不受那次文件损毁影响。

## 7.5 kimi-k2.6:第三种习惯

同款单发格式消融(baseline50,同 48 题):

| 格式 | kimi 正确率 |
|---|---|
| 绝对像素 | 19% |
| **{x,y} [0,1] 浮点分数** | **60%(最优)** |
| {x,y} 整数 [0,1000] | 44% |
| point_2d 整数 [0,1000] | 36% |

kimi 既不是 GPT 的绝对像素、也不是 qwen 的 [0,1000]+point_2d,而是**第三种:{x,y} [0,1] 浮点
分数**。旁证:即使明确要求 [0,1000] 整数或 point_2d,kimi 的原始输出仍频繁滑向 [0,1] 分数、且
JSON 常有畸形(缺引号、像素与分数混装,如 `{"x": 162, "0.591", "y": 0.591}`)——这种"漂移"本身
就是证据:[0,1] 才是它训练里更根深蒂固的习惯。**kimi 的 pipeline(native vs harness)尚未做配对
验证**——kimi 自己的 harness 跑分早早被停(资源让位 qwen),跟 baseline50 重合样本不足以配对
对比,故 `model_profiles.py` 中标记为 `confirmed=False`(provisional,暂按 native 假设)。

三家坐标格式无一相同,通则升级为:**新模型接入前必须实测格式消融,不能用其他模型的结论套用。**

## 8. 框架改造(2026-07)

发现了三种互不相同、有时互相矛盾的模型习惯后,不再用散落各处的 ad-hoc probe 脚本各写一套
prompt/parse 逻辑,而是抽成三层:

1. **`gui_harness/planning/coord_formats.py`** — 坐标格式的唯一真源。每种格式(`abs_pixel` /
   `frac01` / `xy1000` / `point2d_1000`)各有一个 prompt 片段生成函数和一个解析函数。解析对
   "模型漂移"做了鲁棒兜底(如 kimi 的畸形 JSON),这套启发式来自三次消融里反复验证过的行为,
   不是新猜的。**`abs_pixel` 刻意不做 ≤1000→÷1000 的重解释**——4K 截图上一个真实像素坐标完全
   可能小于 1000(比如屏幕左侧 x=519),误重解释会把合法坐标搞错。
2. **`benchmarks/screenspot_pro/model_profiles.py`** — 每模型一条 `ModelProfile`(provider /
   coord_format / pipeline / confirmed / hires / use_hints / note),查表决定策略,不再猜测或
   硬编码 if/else。`confirmed=False` 显式标出"仅测过格式、未测过 pipeline"的模型(目前是 kimi)。
3. **`benchmarks/screenspot_pro/runners/run_sspro_native.py`** — 统一的 native-single-shot 跑分器,
   完全由 profile 驱动,取代了 `probe_qwen_official.py` / `probe_qwen_xy1000.py` /

### 三探针诊断法(正式版)

判定一个模型的坐标 grounding 是"训练出的专用预测头"还是"通用视觉推理能力",复用已有机制、
不需要新造轮子:

- **探针 A(thinking 开关)**:同题 thinking 开/关。推理型模型应更准或持平;预测头模型会变差
  (想得越久,越偏离已校准好的直接映射)。
- **探针 B(母语格式下加提示)**:模型自己的最优格式下,注入 OCR/检测器候选证据。推理型模型应
  更准(真的在权衡证据);预测头模型会变差(证据打断训练好的直通路径)。
- **探针 C(陌生格式下重做 B)**:把模型逼到没专训过的格式,重测提示效应。推理型模型效应符号
  不变;预测头模型会**反转**(被逼出专用通路、退回通用推理模式,这时证据才真的有用)。C 是最
  有力的一条,反转现象没法用"模型笨"来解释。

**跨模型矩阵:**

| 模型 | A · thinking | B · 母语+提示 | C · 陌生格式+提示 | 判定 |
|---|---|---|---|---|
| qwen3.7-plus | 开更差(64%→45%) | −17pt(77%→60%) | **反转 +26pt**(16%→42%) | **纯专训头** |
| kimi-k2.6 | **+8pt**(有效题58%→66%,但thinking开有12%彻底答不出来,失败算错则打平) | **−23pt**(60%→38%,比qwen还负) | 几乎不反转(21%→23%,+3pt) | **部分专训,中间态** |
| MiniMax-M3 | 端点无 thinking 开关,测不了 | **+10pt**(24%→35%) | **+20pt**(0%→20%,同向不反转) | **通用推理型**,我们方法有空间 |
| GPT-5.5 | +2pt(60%→62%,基本打平,慢1.9倍无实质收益) | +8pt(66%→74%)⚠️见脚注 | **+8pt**(50%→58%,同向不反转) | **通用推理型**,与M3同一签名 |

⚠️ GPT 的 B 探针有一个小出入:`probe_gpt_hints_reversal.py` 测出无提示基线 66%,但更早的
`probe_ablation_gpt.py`(同一批 baseline50、同一套 create_runtime+openai-codex 调用)测出无提示
58%——两次"有提示"都收敛在 74%,只有无提示基线不同(29/50 vs 33/50)。同一份代码路径下出现这种
差异,更可能是 GPT 本身的运行间随机性,不是方法论错误。**效应量因此是 +8~+16pt 的区间,但方向
(正、不反转)在两个版本下都成立**,不影响诊断结论。

kimi 是最有意思的反例:B 探针跟 qwen 同方向、甚至更负,但 C 探针没有复现 qwen 那种戏剧性反转。
A 探针最初测成"打平"(27/50=54%两边一样),但修复了两个基础设施 bug(413请求体过大是压缩阈值
踩空导致、与thinking无关;6个ReadTimeout降到150秒重试后**依然全部真实超时**,不是运气差)后
发现:有效题准确率其实是 thinking开(66%)> thinking关(58%),**方向和 qwen 相反**——只是
thinking 开启后 12% 的题直接彻底答不出来,这个可靠性代价刚好抵消了准确率的提升,生产口径下
(答不出来算错)才显得"打平"。**这更像"尝试推理、有时确实更准,但可靠性差",不是 qwen 那种
"推理主动伤害一个已校准好的捷径"**。**说明"专训程度"更像一个连续谱,不是非黑即白的二元分类**
——这本身就是一个可以写进论文的发现。见 `fix_kimi_thinking_errors.py`。

M3 则是三探针里最干净的"通用推理型"案例:B、C 两个条件下证据都是正贡献,且不反转——跟 qwen
(母语负、陌生正、方向翻转)正好相反的签名。这跟 M3 单发天花板低(27%)、高度依赖 harness
(47.4% vs 26%)完全吻合:**M3 没有走 qwen 那条训练记忆的捷径,它的坐标输出是在真的做视觉推理,
只是能力还不够强——这正是我们的方法应该优先投入的对象。** 探针:`probe_kimi_hints_reversal.py` /
`probe_m3_hints_reversal.py`。

### 一个意外的旁支发现:openprogram 的异步资源泄漏

M3/GPT 走的 `Runtime.exec()` 路径(openprogram 主仓库 `agentic_programming/runtime.py`)每次
调用都用 `asyncio.run()` 开一个全新事件循环,但共享的 httpx `AsyncClient` 缓存没跟着清理,
导致每次调用后清理失败(`RuntimeError: Event loop is closed`)、连接漏掉,调用越多内存涨得越多。
qwen/kimi 走的是本仓库自己写的同步 `httpx.Client()`,不碰 asyncio,不受影响。已在 openprogram
主仓库修复(commit `1f501408`,`_reap_loop_clients()`),验证修复后 M3 全程 0 次报错、内存不再
单调下降。这不是本仓库的 bug,但会影响任何走 `Runtime.exec()` 的高频调用场景。

## 9.7 设计级消融:三个设计各值几分(GPT-5.5,2026-07)

论文把 harness 讲成三个设计:①坐标注入(prime)、②自适应粗到细裁剪(adaptive)、
③视觉验证(verify)。在 SSPro-300 分层切片上,从 `sspro_stack_zoom.yaml`(全开,88.7%)
每次整块抽掉一个设计,thinking off,与 single-locate 参考臂(78.3%)对照:

| 臂 | 准确率 | Δ vs full | 中位耗时 | 中位轮数 |
|---|---|---|---|---|
| full(①②③) | **88.7%**(266/300) | — | 114s | 3 |
| −①(`abl_no_prime`) | 87.7%(263/300) | −1.0pt | 134s | 3 |
| −②(`abl_no_adaptive`) | 85.0%(255/300)* | **−3.7pt** | 81s | 1 |
| −③(`abl_no_verify`) | 87.0%(261/300) | −1.7pt | 79s | 3 |
| single-locate(参考) | 78.3%(235/300) | −10.4pt | 25s | — |

**显著性(配对 McNemar,五臂同一批 300 题)**:−② p=0.019(显著)、full vs single
p<0.0001(极显著);−① p=0.63、−③ p=0.33(各自 1~2pt 在 300 题上是 3~5 道题的差距,
**未过显著性**)。所以本节严格证明的是②和整套方法;①的独立效果引单发消融(+16pt,§4),
①③的臂级显著性预期由 M3 跨模型矩阵补上(M3 单发弱、边际空间大)。

**四个发现:**

1. **三个设计抽谁都掉分**——对推理型模型每个都必要;②贡献最大(−3.7pt)。
2. **②吸收了①**:单发时坐标提示值 +16pt(§4),但完整管线里抽掉①只 −1.0pt。
   zoom 把目标放大到看得清之后,模型靠自己的视觉推理就能定位,不再依赖文字坐标兜底。
   ①和②是"补空间信息"的两条冗余路径:单发场景①关键,多轮场景②更强并覆盖①。
3. **剪一刀不等于自适应**:−②仍然剪一次,但单轮只能把 4K 全图切到中位 50% 面积
   (8 轮时最终裁剪框中位面积是全图 1.3%),小目标依旧太小。它 85.0% 落在
   single(78.3%)与 full(88.7%)之间——放大有一点用,但没有"模型自选窗口+多轮收窄
   +错了重选"就吃不满。这直接呼应 C2:固定式 refinement 不如自适应。
4. **成本结构**:③每题花 ~35s(79→114s)买 1.7pt;抽掉①反而更慢(134s)——没有
   候选提示时模型要更多次裁剪尝试才收敛。①是"又快又省"的组件,②是"贵但值",
   ③是"中等价格的保险"。

**数据可靠性(本节自己的审计)**:−②臂首轮跑出 76.3% 是**假数**——31/300 行是本机

### 谜底:传输通道

6月的 claude-code provider 走 **Meridian 代理 → Claude Code SDK**,不是直连 API。
把 claude.exe 当通道重测单发(baseline50,abs_pixel,同 prompt):

| 通道 | 单发 |
|---|---|
| 直连 API,原图直塞 | 30% |
| 直连 + CC 元数据协议(复刻) | 44% |
| 直连 + CC 协议 + [1m] | 48% |

| **Claude Code CLI 通道** | **68%** |

**Claude Code 的图片管线设计(行为测绘 + 二进制字符串确认)**:Read 工具读图时长边
>2000 就等比缩到 2000(sharp,fit:inside,只缩不放),并注入一行

image.]`;≤2000 原样直传无注记;图以工具结果身份进入对话——与 computer-use 训练分布
严格一致。剩余 ~20pt(48→68)在系统提示词/工具结果呈现里,裸 API 复刻不到。

### 修复(全部入库,GPT 路径字节不变)

   坐标的习惯自动归一,scale=1 永不触发)。
3. 发现并绕过 `iterative_min_scale: 1.0` 静默否决 max_side 缩小的陷阱
