ScreenSpot-Pro · GUI Grounding 研究笔记 · 完整版

视觉定位有两种能力:可泛化的通用推理,与格式绑定的特化技能

四个模型(GPT-5.5、qwen3.7-plus、kimi-k2.6、MiniMax-M3)在同一 benchmark 上的对比揭示: 单发分数高不代表真正"看懂"了画面——一部分模型的坐标输出是监督训练出的格式绑定捷径, 本质上是一次前向直接映射的坐标回归头,不依赖推理也无法被推理改进。我们的方法(证据脚手架+ 迭代裁剪)瞄准的是可以被持续放大的通用视觉推理能力;本文的三探针诊断法, 就是用来把这两种能力从表面分数里剥离出来的。本页是 COORDINATE_FORMAT_FINDINGS.md 的完整可视化版本,章节编号一一对应。

2026-07 ScreenSpot-Pro · 1581 样本 判分:预测点落入 GT 框,无 IoU baseline50 / qwen504 / 全量1581 三档评测子集
核心发现

坐标格式敏感性本身就是一个诊断信号:一个模型如果只在某一种坐标约定下"开窍",换个等价的表达 方式就崩到接近 0,说明它在做的不是"理解画面、定位元素"这件通用的事,而是命中了训练时记住的 一种输出习惯。真正值得投入工程精力的是那种不挑格式、能被证据和多轮修正持续改进的通用 视觉推理能力——这正是我们的方法(证据脚手架 + 迭代裁剪)所放大的目标。本文提出的 三探针诊断法(thinking 敏感性、证据敏感性、证据符号一致性)加上一个更直接的格式鲁棒性 指标(同一模型跨4种坐标格式的原始准确率方差,见 §2.5)能在一个模型身上把这两种能力 区分开来,而不只是笼统地问"这个模型准不准"。

01 缘起

qwen3.7-plus 套本 harness(迭代缩放 sspro_stack_zoom.yaml)在 SSPro 全量 62.9%, 明显低于 GPT-5.5 同 harness 的 87.9%,也低于 Qwen 官方自报的 79%。 逐步排查发现根因是坐标格式,而非模型能力、也非分辨率。

单看这一组数字,容易得出"qwen 更强、脚手架反而拖后腿"的印象——但 §05.5/§05.6 会证明这不是 "脚手架没用",而是脚手架在正确地暴露一件事:qwen 这 78% 不是靠理解画面得到的,是命中了 训练时记住的一种坐标输出习惯。它不接受证据、不接受复核、换个等价表达就崩掉——这些都是 "捷径"而非"能力"的特征。我们的方法无法、也不应该去优化一个不基于推理的捷径;它面向的是接下来 在 GPT / M3 上才能看到的那种可持续放大的通用能力。

05.6 为什么会这样:专训预测头 vs 通用推理(战略解释)

假设:qwen 的 point2d_1000 grounding 是监督微调出的专用能力,更接近 "坐标回归头"而非"靠推理解题"。三条独立证据支持:

  1. thinking 开关是反的——若靠推理,想得越久该越准;实测相反(§03.5),说明不是靠推理链条完成的。
  2. "再想想"类操作全部负(候选证据、二次精修,五个独立实验,§05.5)——没法跟一个回归头"讲道理"。
  3. 逼它用没专训过的格式,提示效应直接反转:
坐标格式(离专训约定的距离)无提示有提示提示效应
abs_pixel(完全没专训过)16%42%+26pt
frac01(半对齐)54%60%+6pt
point2d_1000(专训原生格式)73~77%60~65%−12~−17pt

格式离专训约定越远,提示越正向;越接近,提示越负向——几乎单调。这条曲线本身就是最直接的证据: 提示/证据只在模型被迫依赖通用推理时才有用,一旦命中它记忆好的捷径,证据反而是噪声。

战略结论

视觉定位存在两种性质完全不同的能力:一种是监督训练出的格式绑定特化技能 (qwen 的 point2d_1000)——一次前向直接映射,天花板由训练数据分布固定死, 换一种等价表达就近乎失效,工程手段拿不动;另一种是通用视觉推理能力 (GPT 展现出的模式)——能被证据、多轮修正、脚手架持续放大,且不挑坐标表达方式。 我们的方法要挖掘和放大的是后者——这才是能迁移到新应用、新界面、新指令的 真实能力,而不是在某个 benchmark 的坐标约定上"背对了答案"。qwen 单发领先 harness 这件事, 与其说是"qwen 更强",不如说是这道题对它而言不再是一个视觉推理问题——我们的 方法自然帮不上忙,这恰恰是诊断法要识别出来的信号,而不是方法的短板。未来工程精力应该投在 像 GPT、M3 这样能被我们的方法持续拉高的通用推理型模型上。

⚠️ 数据存疑

GPT 单发 78.3%(300题旧数据)grounding_type=direct_pixel,追到代码是旧版 active_loop 候选池机制的兜底分支,不是纯净单发调用,产出它的确切旧代码路径已找不到对应版本。 GPT harness 88.7% 本身干净(§09 核实),不受影响;但"harness 净赚 +10.4pt"这个具体差值精确度存疑。

07 关于 79% 的出处(诚实说明)

Qwen 官方自报 ScreenSpot-Pro 79.0(thinking 关闭、方法未公开)。llm-stats 榜列 qwen3.7-plus 0.790, 但标注"23 self-reported results, 0 verified"——全部厂商自报、零第三方复现 (Claude 0.879 / GPT 0.863 同样自报)。我们用官方格式在 baseline50 上独立复现到 79%, 之前的 52–60% 纯属格式错配。

原始文件 qwen_official.jsonl 后来被一次 sed 脚本误操作覆盖损毁,无法重新核验;但同配方 后续独立重跑两次(77%/48题、73%/50题)落在同区间,结论未受影响。

08 框架改造(2026-07)

发现三种互不相同的模型习惯后,抽成三层,不再用散落各处的 ad-hoc probe 脚本:

  • coord_formats.py 坐标格式唯一真源,每种格式一个 prompt 生成函数 + 一个解析函数,含模型漂移的鲁棒兜底
  • model_profiles.py 每模型一条 ModelProfile(provider/coord_format/pipeline/confirmed/hires/use_hints/note),查表决定策略
  • run_sspro_native.py 统一 native-single-shot 跑分器,profile 驱动,取代重复的一次性脚本

新模型接入 SOP:格式消融(~50题×3-4格式)→ native vs harness 配对(~100+题)→ 填一行 profile,不许假设。

09.5 ⚠️ 更根本的缺口:87.9% 和 88.7% 不是同一套配置

容易反复忘记,务必记住
  • 87.9%(GPT-5.5,全量1581题)—— legacy_baseline.yaml(旧基线,fill缩放,无candidate_sort,8轮)。全量,但配置过时。
  • 88.7%(GPT-5.5,300题子集)—— sspro_stack_zoom.yaml(当前生产配置,M3/qwen全量也用这套)。当前配置,但只有300/1581题。

"GPT 用当前配置、全量 1581"这个数字从来不存在——"87.9%(同配置)/88.7%"的并列说法不准确, 两个数字配置不同、规模也不同。candidate_sort=relevance 是唯一确认稳定 正收益的改进,真实全量数大概率 ≥87.9%,但没有实测过,只是推断。要坐实需 GPT 跑满 1581 题(约 1581×~16次调用),截至目前尚未执行。

三探针诊断法(正式版)

判定一个模型的坐标 grounding 是"训练出的专用预测头"还是"通用视觉推理能力",复用已有机制:

A · thinking 开关→ B · 母语格式下加提示→ C · 换成陌生格式重测 B
探针 A 开 thinking 让模型显式推理更久。推理型应更准或持平;预测头会变差(想得越久,越偏离已校准的直接映射)。
探针 B 母语格式下注入 OCR/检测候选证据。推理型应更准;预测头会变差(证据打断训练好的直通路径)。
探针 C 逼到没专训过的格式,重做 B。推理型效应符号不变;预测头会反转(被逼出专用通路,退回通用推理,证据才有用)。最有力的一条,反转没法用"模型笨"解释。

一个意外的旁支发现:openprogram 的异步资源泄漏

M3/GPT 走的 Runtime.exec() 路径(openprogram 主仓库)每次调用都用 asyncio.run() 开一个全新事件循环,但共享的 httpx AsyncClient 缓存没跟着清理,导致每次调用后清理失败 (RuntimeError: Event loop is closed)、连接漏掉,调用越多内存涨得越多。qwen/kimi 走的是 本仓库自己写的同步 httpx.Client(),不受影响。已在 openprogram 主仓库修复 (commit 1f501408,_reap_loop_clients()),验证后 M3 全程 0 次报错、内存不再单调下降。
